# AI Flow Admin — 架构演进计划

## 1. 当前项目定位

当前项目已具备完整的 **UI 架子能力**，可以稳定支持：

- 样式开发
- 原型搭建
- 视觉走查
- 组件复用

当前项目**暂不以真实业务闭环为目标**，因此以下能力可后置：上传、权限点、动态路由、CRUD 模板、错误边界精细化、打包优化等。

---

## 2. 已完成能力（基线）

### 2.1 工程与分层

- `types -> api -> hooks -> features` 分层已形成
- 请求层使用 `axios`，已具备 token 注入和统一错误转换
- 状态层使用 `zustand`（auth/theme）
- 数据层使用 `react-query`

### 2.2 UI 与样式体系

- `shadcn/ui + Tailwind v4 + token` 已打通
- 语义 class 可直接用于页面开发
- 组件 demo 可作为业务开发模板

### 2.3 表格能力

- `DataTable` 已支持排序、筛选、分页、列显隐、行选择
- 已支持列拖拽排序和列宽拖动

---

## 3. 演进原则（避免过度设计）

1. **按触发条件实施**：不提前做未来 2 个月用不到的能力。
2. **优先薄封装**：图表/上传等先做统一入口，不做重平台。
3. **一阶段一目标**：每阶段有明确产出，确保可验收。
4. **可回滚可替换**：先保证业务可跑，再逐步增强。

---

## 4. 阶段化计划

## 阶段 A：UI 冻结前（当前阶段，持续进行）

### 触发条件

- 仍以“画界面/原型交付”为主
- 未开始真实接口联调

### 目标

- 保持 UI 交付效率
- 降低后续接接口改造成本

### 工作项

1. 页面骨架统一（列表页、详情页、表单页）
2. 组件使用规范固化（优先 ui 组件，减少页面自定义重复）
3. 文档统一（README + 使用文档 + PROJECT）
4. 图表方案定型（选型，不强制全量封装）

### 产出

- 可复制页面模板（最少 2 套）
- 组件使用规范文档 1 份
- 图表选型结论 1 份

---

## 阶段 B：业务对接启动（立即补）

### 触发条件

- 开始调真实接口
- 开始真实表单提交
- 出现文件上传需求

### 目标

- 建立统一业务基础层，避免“每个页面各写一套”

### 工作项（详细）

1. 通用 hooks（`usehooks-ts`）
   - `useDebounceValue`、`useLocalStorage`、`useBoolean` 等
   - 统一搜索防抖、UI 状态控制

2. 通用工具（`lodash-es`）
   - `cloneDeep`、`groupBy`、`throttle` 等
   - 统一数据处理工具入口，避免散落工具函数

3. 上传与预览能力
   - 上传：`react-dropzone` + `browser-image-compression`
   - 预览：`yet-another-react-lightbox`
   - 形成统一 `UploadField` 组件（支持图片压缩、数量限制、回显）

4. 错误处理统一化
   - 在现有 `src/api/client.ts` 基础上补 `useApiError`
   - 统一错误文案映射（网络错误、权限错误、业务错误）
   - 统一 toast 行为，减少重复 `try/catch`

5. 鉴权闭环（建议同阶段完成）
   - 登录页从 mock 切换真实 `useLogin`
   - token 续期策略（refresh + retry）

### 产出

- `hooks/common/*` 通用 hooks 模块
- `lib/common/*` 工具模块
- `components/business/upload/*` 上传组件
- `hooks/useApiError.ts`
- 鉴权闭环文档与流程图

---

## 阶段 C：功能模块扩展（规模化开发）

### 触发条件

- 出现第 2~3 个 CRUD 业务模块
- 开始有权限角色区分
- 菜单数量增长（6~8+）

### 目标

- 建立“批量复制可用”的业务研发模型

### 工作项（详细）

1. 通用 CRUD 页面脚手架
   - 标准布局：查询区 + 表格区 + 操作区 + 弹窗表单
   - 标准行为：分页、筛选、批量操作、状态切换
   - 将 DataTable 作为底座，业务只填列配置和操作回调

2. 权限体系（按后端权限码）
   - 菜单级权限
   - 按钮级权限（如新增/删除/导出）
   - 页面内模块级权限（可选）

3. 路由与菜单配置中心
   - 路由与菜单同源配置，避免维护两套
   - 为后续动态路由（后端返回菜单）预留映射层

4. 存储持久化策略
   - 基于 `zustand/middleware/persist` 统一缓存策略
   - 明确“可持久化字段”和“禁止持久化字段”

5. 图表薄封装（建议此阶段落地）
   - `ChartContainer` 统一 loading/empty/error/theme
   - 页面仅关注业务 option 数据

### 产出

- CRUD 模板页 1 套
- 权限组件/指令 1 套
- 路由菜单配置中心 1 套
- 图表容器组件 1 套

---

## 阶段 D：动态化与性能优化（中后期）

### 触发条件

- 后端提供动态菜单/动态路由
- 页面数量快速增长
- 首屏或列表性能出现瓶颈

### 目标

- 提升系统可扩展性与运行效率

### 工作项

1. 动态路由接入与权限联动
2. 大页面与大组件按路由/模块拆包
3. 大列表渲染优化（必要时虚拟滚动）
4. 资源加载与缓存策略优化

---

## 5. 不在当前计划内（明确延后）

按当前项目决策，以下不作为近期硬目标：

- 全量自动化测试体系建设
- CI/CD 流水线建设

说明：上述能力可在项目进入“多人并行 + 高频发布”阶段后再排期。

---

## 6. 阶段总览

| 阶段 | 目标 |
|---|---|
| A | UI 阶段规范与模板沉淀 |
| B | 业务接入基础设施（hooks/工具/上传/错误/鉴权） |
| C | 规模化开发能力（CRUD/权限/配置中心/图表薄封装） |
| D | 动态化与性能优化 |

---

## 7. 架构负责人的执行清单

1. 每阶段开始前确认触发条件是否满足。
2. 每阶段结束给出“可验收产出”清单。
3. 避免一次性大重构，优先增量替换。
4. 文档与代码同步更新，保持新人可落地。
5. 每周固定 0.5 天做架构债务清理（命名、重复逻辑、组件边界）。
